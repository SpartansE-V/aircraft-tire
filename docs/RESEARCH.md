# Research Findings

> From 5 parallel specialist agents (research phase of the pipeline).

## Aviation maintenance & aircraft-tire domain research for a tire Remaining-Useful-Life (RUL) predictive-maintenance POC

**Key findings**

- WEAR IS CYCLE-DRIVEN, NOT TIME-DRIVEN. Aircraft tires are consumables retired primarily by tread wear from landing/braking scrub, not calendar age. Goodyear states plainly that age is NOT an indicator of serviceability ('no expiration date' if service/visual criteria are met). This means the RUL target variable should be measured in landings/cycles-to-limit (or tread-depth-to-base), and the model is fundamentally a cycle-consumption / wear-rate problem, not a survival-by-age problem.
- TYPICAL LIFE PER TREAD IS ~150-350 LANDINGS. Main-gear airliner tires commonly deliver ~200-300 landings per tread life (industry average ~200-270 takeoff/landing cycles); regional/turboprop 150-400 depending on route and braking; nose tires often 200-350 cycles with different (steering/scrub) stress. Short-haul, high-cycle regional fleets replace far more often than long-haul. This wide, route-dependent spread is exactly the variance a model can exploit.
- THE CASING OUTLIVES THE TREAD: RETREAD CYCLES MATTER. A single tire casing is typically retreaded 5-7+ times before final scrappage; each retread is tracked as a 'retread level' (R-level) marked on the shoulder with retread plant, date, and AEA code. Bias-ply casings dominate aviation partly because of durability and retreadability; radials give more landings-per-tread and run cooler. For the POC the tracked asset is really the CASING (by serial), and RUL should be modeled per tread-life while retaining casing history (R-level correlates with reduced remaining potential).
- CONCRETE WEAR / REMOVAL LIMITS EXIST AND ARE MEASURABLE (good labels). Primary removal trigger: tread worn to the BASE of any groove at any spot, OR to the minimum molded skid depth in the AMM. New-to-worn tread swing is on the order of ~12 mm new down to ~2 mm worn on a representative design. 'Return-to-base' concession: a tire may fly to a maintenance base with top ply cord visible only if cord is not worn through/exposed >1/8 of circumference and not >1 inch wide at the fastest-wearing location. Other hard removal criteria: bulges (separation), sidewall cracks/cuts to casing ply, groove cracking exposing fabric, radial sidewall indentation >=3 mm, rim slip >20 degrees. These give clean, quantitative wear labels for supervised learning and for a computer-vision tread-scoring model.
- INFLATION PRESSURE IS THE #1 CONTROLLABLE LIFE FACTOR, WITH A QUANTIFIED ACTION LADDER. Goodyear/FAA: correct inflation is 'the most important factor' in tire life. Operating band is -0/+5% of specified pressure. Documented cold-pressure action ladder (as % of loaded service pressure): 95-100% reinflate; 90-95% inspect + reinflate + remove if loss >5% recurs in 24h; 80-90% REMOVE tire; <80% remove tire AND its mate (mate was overloaded); 0% scrap tire and mate. A tire/wheel assembly can normally lose up to 5% pressure per 24h. Underinflation causes uneven wear + flex heating (bead area up to 50% hotter) and is most dangerous at takeoff. This ladder is a ready-made deterministic risk-alert rule set to ship alongside the ML model.
- HEAT AND OVERLOAD ACCELERATE WEAR/FAILURE; SPECIFIC TRIGGERS ARE DOCUMENTED. Accelerating factors, several quantified: temperature (tires can run >200F/93C above ambient in operation; 1% pressure change per 5F/3C, so hot/cold airports change effective inflation), long/fast taxi (heat buildup - the dominant fatigue driver at high ground speeds), hard landings, high-energy rejected takeoffs and overspeed (above-rating) landings (MUST be removed/scrapped even if visually fine - internal damage), fuse-plug/thermal release, FOD from poor runway/ramp surfaces (the most common cause of EARLY removal), underinflation, overload/unequal dual pressures (unequal duals make one tire carry extra load), and crosswind/gear-misalignment scrub (causes uneven one-sided wear, sometimes fixable by turning the tire around). These map directly to model features if the corresponding operational data is captured.
- RUL HAS TWO CONCRETE, MODELABLE DEFINITIONS. (1) Cycle-based RUL = estimated landings remaining until tread reaches base-of-groove / minimum skid depth, derived from current tread depth + a per-tire wear rate (mm/landing) that itself depends on route mix, weight, braking energy, inflation history. (2) Depth-based RUL = current minimum groove depth minus limit depth, i.e. remaining wear headroom in mm. 'Wear-to-limit date' (a required POC capability) = RUL-in-cycles / fleet-average cycles-per-day for that tail. Manufacturers already frame this via Remaining Casing Potential (RCP) - additional tread life beyond wear-out - showing the industry thinks in remaining-potential terms.
- INSPECTION DATA IS RECORDED BUT UNDERUSED AND MESSY - THE CORE OPPORTUNITY. Practice: pressure checked cold daily / before first flight; tread and condition inspected each turnaround or after every landing on some aircraft; mandatory thorough inspection after any hard landing. Records (per 14 CFR 43.9/43.11-style entries) capture P/N, S/N, wheel position, removal reason, retread level, and pressure-loss events in the logbook. But historically these are free-text logbook/paper entries - exactly the 'underused historical inspection records' the requirement calls out. The POC's realistic wedge is structuring and mining this existing inspection/removal history, not deploying new sensors.
- SENSOR DATA IS ARRIVING AND SOON MANDATORY (future-proofs the POC). Modern aircraft increasingly carry integrated Tire Pressure Monitoring Systems (TPMS) + brake temperature monitoring (Crane, Safran, etc.), digitizing pressure/temperature per wheel across the axle to the flight deck or ground maintenance. EASA CS-26 (Issue 4) and NPA 2020-05 now push large-aeroplane tyre pressure monitoring so no tyre operates below minimum serviceable pressure - meaning per-flight per-wheel pressure logs will become standard fleet data. FDR/QAR already capture landing vertical acceleration (hard-landing g), aircraft weight, and airport pairs, so hard-landing and load features are obtainable from existing flight data even before TPMS retrofit.
- MODEL FAMILY IS WELL-PRECEDENTED. Tire-wear/RUL prediction via ML is established: image-based CNN tread-depth estimation (contactless), and usage-history regression models claiming high per-position wear-rate accuracy. Aircraft-specific work frames RUL via wear-indicator tracking and Remaining Casing Potential. Analogous prognostics (turbofan RUL, railway wheel tread depth) validate the time-series + degradation-curve approach. For a demo POC, a wear-rate regression + degradation-curve extrapolation is more defensible and data-frugal than deep sequence models.

**Recommendations**

- FRAME RUL AS A WEAR-RATE / DEGRADATION-CURVE PROBLEM, NOT AGE-SURVIVAL. Model target = landings-remaining-to-limit. Core approach for the demo: estimate a per-tire wear rate r (mm tread lost per landing) from history, then RUL_cycles = (current_min_groove_depth - limit_depth) / r. Make r a function of route/weight/braking/inflation features via a gradient-boosted regressor. This is interpretable, trains on modest data, and directly yields all five required capabilities (RUL, wear-to-limit date, prioritization by lowest RUL, spares planning by aggregate wear-to-limit dates, and threshold-based alerts).
- SHIP THE DETERMINISTIC INFLATION ACTION-LADDER AS DAY-ONE RISK ALERTS. Even before any ML converges, implement the documented cold-pressure ladder (-0/+5% band; 90-95% inspect, 80-90% remove, <80% remove tire+mate, >5%/24h recurring loss = remove) and hard-removal flags (worn to groove base, bulge, cord exposed beyond return-to-base limits, post-hard-landing/high-energy-RTO). This gives the demo credible, standards-grounded 'risk alerts for operational disruption' (capability 5) that a maintenance audience will immediately trust, and de-risks the demo if the ML story is thin.
- TRACK THE CASING BY SERIAL AS THE PRIMARY ASSET; MODEL PER TREAD-LIFE. Key every record on tire serial number (Goodyear 8-char YJJJNNNN encodes plant+manufacture date) + retread level (R-level) + current wheel position + gear/axle. Model RUL for the current tread-life but carry casing cumulative-landings and R-level as features (higher R-level ~ lower remaining potential). This mirrors how the industry actually manages the asset and makes spares/inventory planning (capability 4) meaningful (a retreadable casing returned vs a scrapped casing are different inventory events).
- START WITH INSPECTION-RECORD MINING, ADD SENSORS AS A ROADMAP TIER. Phase 1 (demo-able now): structure historical removal/inspection logs (removal reason, tread depth at removal, cycles-on-tire, position, route) into the schema below and fit the wear-rate model - this directly attacks the stated 'underused historical inspection records' problem with data that already exists. Phase 2 (roadmap): ingest TPMS per-wheel pressure/temperature and FDR/QAR landing-g + weight to sharpen wear-rate features. Present sensors as future accuracy upside, not a demo prerequisite - avoids blocking the POC on hardware you don't have.
- ADD A COMPUTER-VISION TREAD-SCORING DEMO AS THE WOW FACTOR, SCOPED NARROW. A CNN that estimates min-groove-depth / 'distance to base of groove' from a phone photo of the tread is a strong, self-contained demo tile that turns qualitative inspection into a quantitative RUL input and showcases the 'computer-vision inspection' capability. Scope it to depth-band classification (e.g. healthy / monitor / near-limit / at-limit) rather than sub-millimeter regression to keep labeling tractable for a POC.
- PRIORITIZE WHEELS WITH A COMPOSITE RISK SCORE, NOT RUL ALONE. Capability 3 (prioritize wheels) should rank by a blend of: predicted RUL-in-cycles, tail utilization (cycles/day -> soonest wear-to-limit date), current pressure-ladder status, recent hard-landing/high-energy events, and spare availability for that tire size. A pure lowest-RUL sort will bury operationally urgent low-utilization tires; the composite makes the prioritization defensible to planners.
- SEED WITH A REALISTIC SYNTHETIC/SEMI-SYNTHETIC DATASET FOR THE DEMO. Given data-access risk, generate a physics-informed synthetic fleet: sample new tread ~ manufacturer skid depth, draw per-landing wear from route-mix + weight + braking distributions, inject inflation-loss events and hard landings, and retire at base-of-groove. This lets the POC demonstrate the full pipeline (forecast, wear-to-limit date, prioritization, spares, alerts) convincingly while a real historical extract is negotiated, and doubles as a data-schema validation harness.
- CONSTRAIN SCOPE TO ONE FLEET/TIRE SIZE AND MAIN-GEAR FIRST. Wear physics, cycles-per-tread, and removal limits differ by aircraft type, tire size, and position (nose vs main). Fix the POC to a single narrowbody fleet (e.g. A320/737-family) and main-gear tires of one size to keep wear-rate distributions coherent and labels comparable. Expand to nose gear and other fleets only after the single-fleet model is credible.

**Risks**

- DATA AVAILABILITY IS THE #1 RISK. The whole premise rests on historical inspection/removal records that are typically free-text paper/logbook entries with inconsistent tread-depth capture (often only 'worn to limit' at removal, not a depth time-series). Without at least removal-cycle counts + removal reasons per serial, wear-rate estimation is weak. Mitigate with the pressure-ladder rules + synthetic data so the demo still stands.
- LABEL SPARSITY / RIGHT-CENSORING. Tires are removed at a limit, so you observe mostly end-of-life points, not full continuous wear curves; many removals are for FOD/damage (random, not wear) which are censoring events, not wear signal. Must separate wear-out removals from damage/FOD/hard-landing removals or the wear-rate model is biased. Removal-reason coding is therefore essential and often missing.
- FOD AND HARD-LANDING REMOVALS ARE LARGELY UNPREDICTABLE. FOD is the most common cause of EARLY removal and high-energy events force immediate scrapping regardless of wear - these are near-random shocks a wear model cannot forecast. Overstating RUL accuracy for a fleet where a large share of removals are damage-driven will erode trust. Report predicted-wear-out RUL and event-driven risk separately.
- SAFETY-CRITICAL OVER-RELIANCE. A tire blowout at takeoff is catastrophic; the POC must be positioned as decision-support that AUGMENTS mandated daily cold-pressure checks and AMM inspections, never replaces them. Any suggestion the model lets operators skip inspections is a regulatory and safety non-starter (FAA AC 20-97B / EASA CS-26 mandate the checks).
- FLEET/POSITION HETEROGENEITY BREAKS TRANSFER. A model trained on one tire size/aircraft/position will not transfer to another (different skid depth, ply rating, load, cycles-per-tread). Pooling heterogeneous tires without stratification produces meaningless average wear rates. Scope discipline is required.
- INFLATION HISTORY IS RARELY LOGGED AT DEPTH. The most predictive controllable factor (chronic slight underinflation) is exactly what legacy records don't capture continuously - pre-TPMS you get at best a pass/fail cold check, not a pressure time-series. Best wear-rate features may be unavailable until TPMS data exists, capping pre-sensor accuracy.
- TEMPERATURE/ROUTE CONFOUNDING. Hot-airport operations, long taxi, and braking energy jointly drive heat-related wear but are correlated with each other and with airport pairs; naive feature attribution may mislead planners about causes. Keep the demo's causal claims modest.
- CV TREAD MODEL GENERALIZATION. A tread-depth CNN trained on clean demo photos may fail on real ramp conditions (grease, water, chevron-cut runway marks that mimic cuts, variable lighting/angle). Treat CV output as an assistive band estimate with human confirmation, not an authoritative measurement, for the POC.

## Modeling approaches for aircraft tire RUL and wear-to-limit-date forecasting from sparse, irregular inspection records plus operational cycle data (POC scope)

**Key findings**

- DATA REGIME IS THE DECIDING FACTOR. Aircraft tire tread-depth measurements are SPARSE, IRREGULAR, and FRAGMENTARY (a handful of manual gauge readings per tire over its life at unequal intervals), not dense sensor streams. This rules out the deep-learning prognostic playbook (LSTM/Transformer, C-MAPSS-style) as the primary model, because those need long, regularly-sampled multivariate trajectories. The literature explicitly flags that most RUL methods assume 'complete and dense monitoring data' and that sparse/fragment data is the hard, under-served case. The right primary tool is degradation-path modeling with mixed effects, backed by survival analysis.
- SIX RUL PARADIGMS, MAPPED TO FIT. (a) Degradation/health-index modeling (fit a wear curve, extrapolate to the legal tread-depth limit) -> BEST FIT for sparse tread data; directly yields both RUL and a wear-to-limit date. (b) Survival analysis (Weibull AFT, Cox PH) -> strong complement; handles right-censoring (tires still in service = not yet failed) natively, converts cycles-since-new + covariates into a failure-time distribution and confidence bounds. (c) Regression on remaining cycles (GBM/XGBoost, RF) -> good, fast, explainable baseline when you have enough retired-tire examples with known end-of-life. (d) Similarity-based / NASA C-MAPSS trajectory matching -> POOR FIT: needs dense run-to-failure trajectory libraries you will not have from sparse inspections. (e) Time-series forecasting Prophet/ARIMA -> WEAK for per-tire sparse data (too few, unevenly-spaced points per unit; no seasonality); usable only for fleet-level demand/consumption forecasting, not per-wheel wear. (f) LSTM/deep sequence models -> premature for a POC; data-hungry, opaque, needs dense streams.
- THE UNIT OF DEGRADATION SHOULD BE CYCLES (or landings), NOT CALENDAR TIME. Tire wear is driven by takeoff/landing/taxi cycles and load, not wall-clock days. Model wear-per-cycle, then convert to a DATE only at the end using each aircraft's forecast utilization (cycles/day). This decouples the physics (wear vs cycles) from the logistics (calendar planning) and makes the model transferable across aircraft that fly at different rates.
- WEAR-TO-LIMIT DATE = THRESHOLD-CROSSING OF THE EXTRAPOLATED CURVE, WITH UNCERTAINTY. Fit remaining tread depth vs cumulative cycles (typically near-linear per tire, so a random-slope linear mixed-effects / hierarchical Bayesian model works well), extrapolate the per-tire curve down to the serviceable-limit tread depth, and read off the crossing cycle. Wrap it in a prediction interval (delta method, or Monte Carlo / bootstrap over the fitted slope+intercept posterior). Convert the cycle interval to a calendar-date interval using utilization forecast. This is the standard 'first-passage-time to threshold' formulation from degradation-modeling literature and is the single cleanest way to produce a DATE + confidence interval.
- PARTIAL POOLING IS ESSENTIAL FOR SPARSE DATA. With only 2-5 readings on a young tire you cannot fit a reliable per-tire slope in isolation. A hierarchical / mixed-effects model borrows strength from the fleet (a fleet-level mean wear rate + per-tire deviations by position, aircraft type, route/load class). New or barely-measured tires shrink toward the fleet prior; well-measured tires get their own slope. This is exactly the fragment-data remedy the sources describe (mixed-effects and Bayesian semi-parametric residual-life methods).
- SURVIVAL ANALYSIS HANDLES CENSORING AND COVARIATES THAT PURE REGRESSION IGNORES. Most in-service tires are right-censored (not yet removed). Weibull AFT gives an interpretable, parametric time/cycles-to-removal distribution with covariate acceleration factors (wheel position, aircraft type, airport/route harshness, load) and closed-form quantiles for alert thresholds; Cox PH gives covariate hazard ratios without assuming a shape. Use survival as the risk/priority engine and censoring-correct backbone; use the degradation curve as the physical wear predictor. They are complementary, not competing.
- RISK/PRIORITY SCORE SHOULD COMBINE PROBABILITY x CONSEQUENCE. A defensible per-wheel score = P(tire crosses limit before next scheduled maintenance/inspection opportunity) x operational-consequence weight (AOG exposure of that tail, spare availability at that station, wheel position criticality). The probability comes directly from the threshold-crossing interval or the survival curve S(t). Rank all wheels by this score; alert when P(cross before next check) exceeds a tuned threshold (e.g. >20 percent) or when the lower confidence bound of the wear-to-limit date falls inside the planning horizon.
- EVALUATION MUST USE PROGNOSTICS-SPECIFIC METRICS, NOT JUST RMSE. Point-error (RMSE/MAE on RUL-in-cycles) is necessary but insufficient. Add the Saxena et al. suite: Prognostic Horizon (how early do predictions become and stay reliable), alpha-lambda accuracy (does prediction stay within an alpha-band of truth at each life fraction lambda), Relative Accuracy, and Convergence. Critically, use an ASYMMETRIC score that penalizes LATE (over-estimated RUL) predictions more than early ones, exactly like NASA's C-MAPSS scoring (over-estimation is the dangerous direction: a tire that fails before you predicted causes the AOG you are trying to avoid). NASA's PrognosticsMetricsLibrary implements these.

**Recommendations**

- RECOMMENDED POC MODEL (primary): a random-slope hierarchical/mixed-effects linear degradation model of tread_depth ~ cumulative_cycles, with per-tire random intercept+slope partially pooled toward fleet-level fixed effects grouped by wheel position, aircraft type, and a route/load-harshness class. Predict RUL = (current_tread - limit)/estimated_wear_rate; predict wear-to-limit cycle by threshold crossing; convert to a DATE via utilization forecast. This trains in seconds, is fully explainable (a wear rate and an intercept per tire you can plot), naturally handles unequal spacing and few points, and directly yields both required outputs (RUL and wear-to-limit date). Implement with statsmodels MixedLM or a small PyMC/Bayesian model (Bayesian gives cleaner posterior intervals for free).
- RECOMMENDED SECONDARY MODEL (risk engine + censoring): a Weibull Accelerated Failure Time (AFT) survival model on cycles-to-removal with covariates (position, aircraft type, station/route class, load). Use lifelines (WeibullAFTFitter) — a few lines, handles right-censored in-service tires correctly, and outputs a survival curve S(cycles) per wheel plus quantile-based alert thresholds. Cross-check its RUL quantiles against the degradation-curve extrapolation; agreement builds trust in the demo.
- OPTIONAL FAST BASELINE for the demo bake-off: a gradient-boosted regressor (XGBoost/LightGBM) predicting remaining cycles from engineered features (current tread depth, wear rate from last 2 readings, cycles-since-new, position, aircraft type, cumulative load proxy). It gives a strong accuracy anchor and SHAP-based explanations, and exposes whether the linear-wear assumption is leaving signal on the table. Keep it as a comparison, not the headline model, because it cannot extrapolate a wear-to-limit DATE as cleanly as the degradation curve.
- WEAR-TO-LIMIT DATE WITH CI — concrete recipe: (1) fit the mixed-effects/Bayesian model; (2) for each tire draw N samples of (intercept, slope) from the posterior/fitted covariance (or bootstrap residuals); (3) for each draw solve cycle* where tread crosses the serviceable limit; (4) map cycle* to a date via that tail's forecast cycles/day; (5) report the median date plus, e.g., 10th-90th percentile band as the confidence interval. Show the fan chart in the demo — decision-makers respond to 'earliest credible replacement date' (the lower bound) far more than a point estimate.
- PRODUCT/UX for the 5 capabilities: (1) RUL = remaining cycles + equivalent days; (2) wear-to-limit date = median date + earliest-credible date; (3) prioritization = sortable table ranked by risk score = P(cross before next check) x consequence weight, color-coded; (4) inventory/planning = aggregate predicted removals per station per week into a spare-demand forecast (this is where fleet-level time-series like Prophet legitimately helps — forecasting TOTAL tire consumption, not per-wheel wear); (5) risk alerts = fire when lower-bound wear-to-limit date enters the planning window OR P(cross before next scheduled check) exceeds a tuned threshold.
- EVALUATION PLAN for the POC: hold out a set of RETIRED tires with known removal cycle. Report RMSE/MAE on RUL-in-cycles AND on wear-to-limit date; plot alpha-lambda accuracy (alpha=0.2 band) across life fractions; report Prognostic Horizon; and compute an asymmetric score with heavier penalty on late/over-estimated RUL (mirror NASA's asymmetric exponential). Reuse NASA's PrognosticsMetricsLibrary rather than re-deriving. This turns the demo from 'looks plausible' into 'measurably safe-biased'.
- SEQUENCING: Week 1 — assemble tire history table (per reading: tire id, tail, position, aircraft type, date, cumulative cycles, tread depth, removed flag), engineer cycles-since-new + wear-rate features. Week 2 — fit mixed-effects degradation model + Weibull AFT; build threshold-crossing + Monte-Carlo CI. Week 3 — risk score, alerts, spare-demand rollup, evaluation dashboard. Defer LSTM/CV tread-image and dense-sensor ideas to a post-POC roadmap slide.

**Risks**

- SPARSITY / IDENTIFIABILITY: a tire with only one reading has no observable slope. Mitigate with partial pooling (shrink to fleet/position prior) and by refusing to emit a tight CI when data is insufficient — surface 'insufficient data, using fleet prior' rather than a falsely-confident date. This honesty is itself a demo strength.
- LINEAR-WEAR ASSUMPTION MAY BREAK: real tread wear can accelerate near end-of-life or after retreading, and shows regime changes with route/season/load. A pure linear extrapolation will then over-estimate RUL (the dangerous direction). Mitigate by validating linearity on retired-tire curves, allowing a piecewise/2-phase or mild nonlinear (e.g. quadratic) term if residuals show curvature, and biasing alerts conservative.
- SURVIVOR / SELECTION BIAS: inspection cadence is not random — tires flagged as suspect get measured more often, and removed tires stop generating data. This biases naive wear-rate estimates. Handle right-censoring explicitly (survival model) and be cautious treating inspection frequency as informative.
- UTILIZATION FORECAST IS A SECOND SOURCE OF ERROR: the wear-to-limit DATE depends on future cycles/day, which varies with schedule changes, seasonality, and aircraft reassignment. A perfect wear model with a wrong utilization assumption still gives a wrong date. Expose the utilization assumption explicitly and let planners override it; propagate its uncertainty into the date CI.
- OVER-ESTIMATION IS THE COSTLY ERROR: predicting more life than exists causes the exact AOG/emergency-logistics event the project targets. Bake asymmetry into BOTH the loss used for tuning and the alerting thresholds (alert on the lower confidence bound, not the median). Symmetric RMSE-only tuning is a trap here.
- SMALL RETIRED-TIRE SAMPLE for evaluation: if few tires have reached documented end-of-life, prognostic metrics (Prognostic Horizon, alpha-lambda) will be noisy. Supplement with leave-one-tire-out cross-validation and clearly caveat sample size in the demo.
- DATA PLUMBING RISK: cumulative cycles per tire (across tire moves between wheel positions/aircraft, and across retreads) is often the hardest field to reconstruct and is the model's key x-axis. Budget real time for building a clean tire-genealogy / cycles-since-new field; the model quality is capped by it.

## Synthetic data engineering: physics-informed tire-wear generator, entity schema, and supervised RUL training data (censored vs failed) for the aircraft-tire RUL POC

**Key findings**

- No real dataset exists, so the whole POC rests on a defensible synthetic generator. The credible approach is physics-informed: model groove depth as a monotonically decreasing function of accumulated per-landing 'wear increments', where each landing consumes tread as a function of aircraft type, load, landing severity, taxi distance, inflation deviation, and temperature. Real-world numbers anchor the ranges: new main-tire groove depth ~9-14 mm, wear limit ~1.6-2.4 mm (FAA/Goodyear cite 0.79-2.38 mm minimum), and narrow-body main tires last ~150-400 landings (nose tires shorter). These bounds make the generator auditable rather than arbitrary.
- The correct target formulation is groove-depth-based, not a raw cycle count. RUL = (measured_groove_depth - wear_limit) / current_wear_rate_per_cycle, expressed in remaining landing cycles AND a wear-to-limit date (via each tire/route's cycles-per-day rate). This directly serves POC capabilities (1) forecast RUL, (2) predict wear-to-limit dates, and lets the model learn from partial inspection histories rather than needing a tire to be fully worn out.
- Censoring is the central modeling reality and must be baked into the generator, not added later. Three removal outcomes must be simulated: (a) WORN — tire reaches wear limit (uncensored, clean RUL label); (b) IN_SERVICE — tire still mounted at snapshot date (right-censored, groove above limit); (c) EARLY_REMOVAL — FOD/damage/cut removal before wear-out (censored w.r.t. wear-RUL, since it did not fail from wear). Literature confirms FOD is the most common cause of premature removals, so a realistic ~15-30% early-removal fraction is needed or the model will overstate achievable life.
- A tire's wear trajectory is not a single global slope. Each physical tire needs a per-tire random-effect 'wear susceptibility' multiplier (manufacturing/compound variation) plus per-position and per-aircraft-type baselines (nose vs main inboard/outboard wear very differently). Without this hierarchical variation, every serial looks identical and the model cannot learn asset-level prioritization (capability 3).
- Two noise layers are required and must be kept separate: (a) measurement noise on inspection readings (tread-depth gauge error, ~0.2-0.4 mm, plus occasional gross mis-reads / transposed digits) and (b) process noise in the underlying wear (per-landing stochasticity + rare accelerated-wear events from hard landings, under-inflation runs, or a damaged runway). Conflating them makes the true-vs-observed gap impossible to reason about when validating the model.
- Feature engineering on inspection history is where 'historical inspection records are underused' (the stated problem) becomes model signal. The highest-value engineered features are: empirical wear rate (mm lost / cycles between last two inspections), rolling wear slope over a window, cycles-since-install, mean/max landing severity in the interval, inflation-deviation exposure, and days-since-last-inspection (data-recency / staleness signal for alert confidence).

**Recommendations**

- Generate at the landing-event grain first, then aggregate up to inspections. For each tire, simulate a stream of landings (each with its own severity/load/taxi/temp/inflation), compute a per-landing wear increment, and accumulate to get the true groove-depth curve. Then sample inspection dates from that curve and apply measurement noise. This ordering guarantees inspections are physically consistent with cycles and gives you both the observed dataset (what the model sees) and the ground-truth curve (what you validate against).
- Use a multiplicative wear equation so factors compose realistically and stay non-negative: wear_per_landing = base_wear[type,position] * f_load * f_severity * f_taxi * f_inflation * f_temp * tire_susceptibility * process_noise. Keep each factor a dimensionless multiplier centered near 1.0 (e.g. hard landing pushes f_severity to 1.5-3x, under-inflation pushes f_inflation to 1.3-2x). See dataOrTechNotes for the exact form and ranges. This is transparent, tunable, and every coefficient maps to a domain lever a maintenance engineer will recognize.
- Model early removals as a competing risk. For each landing, draw a small FOD/damage hazard (e.g. 0.05-0.15% per landing, elevated on rough/foreign-object runways); if it fires before wear-out, mark the tire EARLY_REMOVAL with an event_type and censor its wear-RUL. This produces a realistic mix of uncensored, right-censored, and competing-risk-censored records that mirrors real fleet data and forces the training pipeline to handle censoring correctly (survival loss, or filtering with explicit censoring flags).
- Store both a `true_groove_depth` (noise-free ground truth) and a `measured_groove_depth` (with gauge noise) column in the synthetic tables, and keep true generator parameters (susceptibility, base rates, event flags) in a hidden `_ground_truth` sidecar not exposed to training. This lets the POC report honest model error against known truth, demo 'the model recovered the wear rate within X%', and debug feature leakage — a huge credibility win for a demo where reviewers will (rightly) ask 'how do you know it works?'.
- Build the supervised training frame as one row per (tire, inspection) 'observation point', with the label = remaining cycles to wear-limit computed from that point forward using the ground-truth curve, and a `censored` boolean + `time_to_event` for the survival-aware variant. Train two heads: a regression head for RUL-in-cycles (drives forecasts and prioritization ranking) and a date projection that multiplies predicted remaining cycles by the wheel's recent cycles-per-day to yield wear-to-limit dates (capability 2) and risk alerts (capability 5).
- Make fleet size and horizon demo-appropriate: e.g. 2-3 aircraft types, ~20-40 aircraft, ~8-10 wheel positions each, 12-24 months of history, inspections every ~15-30 cycles or on a calendar cadence. That yields tens of thousands of landing events and thousands of inspection rows — enough to train and cross-validate a gradient-boosted or simple survival model while staying fast to regenerate. Seed the RNG and version the generator config so every demo run is reproducible.
- Engineer features strictly from data available up to the inspection date (no lookahead) to prevent leakage: wear_rate_recent, wear_slope_rolling(k), cycles_since_install, days_since_last_inspection, mean_landing_severity_window, hard_landing_count_window, inflation_deviation_exposure, position/type one-hots, brand, and groove_depth_remaining_ratio = (measured - limit)/(initial - limit). Validate that model RUL error tracks the ground-truth curve and that prioritization ranking (capability 3) correlates with true remaining life.

**Risks**

- Garbage-physics risk: if coefficients are chosen carelessly the model learns the generator's artifacts, not tire physics, and the demo silently teaches a false story. Mitigation: anchor every parameter to the cited real-world ranges (150-400 landings, 9-14 mm new, ~1.6-2.4 mm limit), sanity-check that median simulated tire life lands in-band, and have a domain reviewer eyeball a sample of wear curves before training.
- Label leakage / too-easy learning: because RUL is derived deterministically from the same wear equation that generated features, a model can overfit to a near-perfect analytic relationship and post unrealistically low error — impressive in the demo, meaningless for real deployment. Mitigation: inject genuine measurement + process noise and early-removal censoring, hold out whole tires/aircraft (not random rows) in CV, and report error under noise, explicitly framing the POC as 'method demonstration on synthetic data, real accuracy TBD'.
- Censoring handled wrong: if IN_SERVICE and EARLY_REMOVAL tires are dropped or treated as if they reached the limit, the training set is biased toward short-lived tires and RUL is systematically underestimated. Mitigation: carry explicit `outcome`/`censored`/`time_to_event` columns end-to-end and use a survival-aware objective or, at minimum, correct filtering with documented assumptions.
- Synthetic-to-real gap is unquantifiable by construction: no held-out real data means we cannot state real-world accuracy. This must be stated plainly to stakeholders so the POC is scoped as a workflow/plumbing and method proof, not an accuracy guarantee. The moment real inspection records arrive, the generator's distributions should be re-fit to them.
- Over-engineering the schema for a demo: modeling every FK relationship, full soft-delete lifecycle, and every operational field can burn the POC timeline. Mitigation: keep the schema flat and denormalized enough to train fast (the schema below is the minimum credible set), and defer production concerns (multi-tenant fleets, audit trails) until after the demo proves value.
- Time-grain explosion: simulating every landing for a large fleet over years can blow up row counts and generation time. Mitigation: cap fleet/horizon to demo scale, generate landing events in vectorized batches, and persist aggregated inspection rows for training while keeping raw landings optional/regenerable.

## Competitive & market landscape for aircraft tire RUL / aviation predictive-maintenance (POC P3)

**Key findings**

- THE SPACE IS NOT GREENFIELD - two tire OEMs already ship aircraft tire life prediction. Bridgestone + Japan Airlines run an algorithmic tire-wear prediction system live since May 2020 (regional J-AIR jets), expanded to A350-900 and large jets in 2024; it converts unscheduled tire changes into scheduled ones, cut/stabilized tire+wheel stock, and reduced maintenance overtime. Michelin + Safran 'PresSense' is a connected tire (RFID pressure sensor in a Michelin tire) that reads pressure ~45 min after landing vs ~3 hrs manually, feeds a smartphone app + database for predictive maintenance; Azul adopted it across ~110 A320/A321/E195 aircraft. These are the true competitors, not the generic fleet-health platforms.
- BUT INCUMBENT SOLUTIONS ARE LOCKED-IN AND NARROW. Bridgestone's model is tied to Bridgestone tires + specific aircraft flight-data feeds (OEM lock-in, per-airline bespoke rollout). Michelin/Safran PresSense is a HARDWARE play centered on tire PRESSURE (a safety/inflation signal), requires sensor-equipped tires + RFID readers, and is pressure-first, not tread-RUL-first. Neither is a vendor-agnostic software layer that ingests an airline's EXISTING historical inspection records across mixed tire brands - which is exactly the underused-data gap P3 targets.
- AIRCRAFT-HEALTH-MONITORING GIANTS DO NOT COVER TIRES. GE Aerospace (SkyEdge/digital twins), Honeywell Forge, Airbus Skywise (w/ Palantir; airlines report up to 15% fewer unscheduled maintenance events), and Boeing AnalytX/Airplane Health Management focus on engines, APUs, avionics and LRUs streamed off aircraft data buses. Tires are NOT instrumented on the data bus and are largely absent from these platforms - a structural blind spot the POC can occupy without competing head-on with these deep-pocketed players.
- MRO SOFTWARE (AMOS/AMOSsmart, TRAX, Ramco) TRACKS TIRES AS COMPONENTS BUT DOES NOT FORECAST RUL. These systems handle serial-number tracking, work orders, compliance and scheduled maintenance; predictive analytics is bolted on and generic. Searches surfaced no dedicated tire-RUL / wear-to-limit forecasting module in AMOS/TRAX/Ramco - they are the integration target (system of record) rather than a competitor for the prediction layer.
- GOODYEAR IS BEHIND ON THE PREDICTIVE-AIRCRAFT-TIRE SPECIFIC ANGLE. Goodyear's predictive-analytics tire management (Proactive Solutions, intelligent tire) is centered on TRUCK/OTR/commercial ground fleets, not a productized aircraft-tire RUL offering. Aircraft tire tread-wear ML/CV research exists broadly (1D-CNN tread-wear models ~4.6-5.2% RMSE; tire-image CNN classifiers ~99.7% accuracy) but is mostly automotive/academic, confirming the modeling approach is feasible and not yet productized for aviation tires by a neutral vendor.
- MARKET DRIVERS ARE STRONG AND QUANTIFIED. AOG costs ~$10-20K/hr (narrowbody) to $30-150K/hr (widebody), up to ~$1.2M/day; cumulative industry AOG impact estimated ~$62B/yr. Predictive maintenance case studies: SmartLynx cut AOG incidents 57% and downtime 49% (~EUR840K/yr saved); Delta cut maintenance-related cancellations 95% and hit >90% parts-demand accuracy (APEX) extending planning to 18 months; Azul pushed AOG prediction windows from 1-2 days to 2-3 weeks. AI spare-parts optimization shows ~20% inventory-cost reduction and ~15% less downtime. Airlines report 30-45% AOG reduction specifically on high-disruption parts including tires, brakes, landing gear.
- TIRES ARE A HIGH-FREQUENCY, HIGH-VOLUME CONSUMABLE - IDEAL FOR RUL FORECASTING. Aircraft tires are typically removed every ~700-800 landings (some as low as ~400); ~90% of commercial aircraft fly on retreads; the aircraft tire retreading market alone is ~1.2M units/yr (2024) growing toward ~1.8M units. Total aircraft tire market ~$1.6B-$2.9B (2024, estimates vary), commercial aviation ~55% of retread demand, Asia-Pacific fastest-growing (~8% CAGR). Wear is driven by variable, hard-to-model factors (hard/cross-wind landings, anti-skid, sidewall deflection, runway condition) - exactly where ML on historical inspection data beats fixed landing-count rules that cause premature removal or late-removal AOG risk.
- REGULATORY/SAFETY TAILWIND ON TIRE CONDITION. NTSB has recommended daily tire-pressure checks and TPMS on all transport-category aircraft after underinflation-caused accidents (all-four-main-gear failures on takeoff); FAA currently accepts 96-hr pressure checks. SAE ARP6137 and EASA rulemaking (NPA 2020-05) address tyre-pressure monitoring. This validates buyer urgency around tire condition, though note it centers on PRESSURE/inflation - leaving TREAD-WEAR RUL + inventory as a distinct, less-crowded problem for the POC.

**Recommendations**

- POSITION AS A VENDOR-AGNOSTIC, SOFTWARE-ONLY 'TIRE RUL & INVENTORY BRAIN' - the explicit anti-lock-in play. Bridgestone locks you to Bridgestone tires + flight data; Michelin/Safran locks you to sensor-equipped tires + RFID hardware. Sell to airlines/MROs that run mixed tire brands and want ONE prediction layer over their EXISTING historical inspection records - no new hardware, no tire-OEM switching cost. This is the single most defensible differentiation angle.
- LEAD WITH THE UNDERUSED-DATA WEDGE, NOT SENSORS. The requirement's core insight - historical inspection records are underused - IS the moat vs sensor-based incumbents. Frame the POC as 'turn the tread-depth/pressure/removal logs you already collect at every check into RUL forecasts and wear-to-limit dates.' This needs zero capex from the buyer and sidesteps the hardware-adoption barrier that slows PresSense-style rollouts.
- TARGET TREAD-WEAR RUL + INVENTORY, NOT PRESSURE. Pressure monitoring is crowded (PresSense, TPMS mandates, SAE ARP6137, patents galore) and increasingly hardware/OEM-owned. The wear-to-limit + spare-inventory + wheel-prioritization + AOG-alert stack (P3's 5 capabilities) is far less productized by neutral vendors. Anchor the demo on 'which wheels to pull next and how many spares to pre-position at which station,' which ties directly to the $10-150K/hr AOG and 20% inventory-cost prizes.
- MAKE THE POC AN MRO-SYSTEM COMPANION, NOT A REPLACEMENT. Explicitly show ingestion from / export to AMOS, TRAX, Ramco (system of record). Airlines will not rip out their MRO ERP; the winning wedge is a prediction layer that reads their component/inspection history and writes prioritized work recommendations back. Positioning as an integration/overlay lowers sales friction and avoids competing with entrenched ERP incumbents.
- QUANTIFY THE DEMO IN THEIR CURRENCY: AOG hours avoided, premature-removal waste avoided, spares pre-positioned. Use credible public benchmarks (AOG $10-150K/hr; 30-45% AOG reduction on high-disruption parts incl. tires; 20% inventory-cost cut; Bridgestone/JAL's 'unscheduled->scheduled' conversion and stock stabilization) to frame ROI in the pilot deck, so a demo-able POC still tells a board-ready value story.
- PICK A NARROW PILOT WEDGE MATCHING P3's DEPLOYMENT FOCUS: one fleet type + one wheel-assembly family + one hub. Tires' 700-800-landing cadence means you accumulate enough removal/inspection events for a credible RUL model quickly (unlike engines). A single-fleet, single-station pilot produces measurable AOG/inventory outcomes within a short window - ideal for a POC-to-production narrative.
- BUILD CV TREAD-INSPECTION AS AN OPTIONAL DATA-ENRICHMENT MODULE, NOT THE CORE. Tire-image CNNs already hit ~99.7% wear classification and thermal/1D-CNN models hit ~4.6-5.2% RMSE, so a phone-photo tread-depth capture can cheaply upgrade sparse manual logs. But keep the core RUL engine runnable on tabular inspection history alone, so buyers with no CV pipeline can still adopt.

**Risks**

- INCUMBENT CREDIBILITY & OEM LOCK-IN. Bridgestone (with JAL, live 4+ years) and Michelin/Safran (with Azul) have physics/materials expertise, direct tire-manufacturing data, and existing airline relationships a startup cannot match. They can bundle prediction 'free' with tire supply contracts, undercutting a standalone software price. Mitigation: win on neutrality/multi-brand + zero-hardware, and target airlines running mixed OEM tires where bundled OEM tools don't cover the whole fleet.
- DATA ACCESS & QUALITY IS THE EXISTENTIAL RISK. The whole thesis depends on historical inspection records that are notoriously sparse, paper-based, inconsistently digitized, and siloed inside MRO ERPs. If the pilot airline's tread-depth/removal logs are thin or unreliable, RUL accuracy collapses. Mitigation: scope the POC to an airline with digitized records; include Bridgestone's 'easytrack' (QR/barcode) style serial tracking as a data-quality prerequisite; be explicit about minimum data requirements.
- TIRE WEAR IS DRIVEN BY HARD-TO-CAPTURE OPERATIONAL VARIABLES. Landings-per-tire varies with hard/cross-wind landings, anti-skid, sidewall deflection and runway condition - many not in inspection logs. Models trained only on tread-depth-over-time may miss regime shifts, producing over-confident RUL. Mitigation: fuse operational proxies (route/station, aircraft, load) where available; present RUL as calibrated probability bands with wear-to-limit DATE RANGES, not false-precision single dates.
- REGULATORY & SAFETY-OF-FLIGHT LIABILITY. Tires are a flight-safety item; a missed under-prediction that contributes to a tire failure/RTO is a catastrophic-liability scenario, and airworthiness authorities (FAA/EASA) constrain how condition-based decisions replace hard-time limits. Mitigation: position POC as DECISION SUPPORT that prioritizes/advises within existing removal limits (never extends beyond them), keeping the human/AMM in the loop - a demo-able POC, explicitly not a certified airworthiness system.
- LONG, CONSERVATIVE AVIATION SALES CYCLES & SWITCHING INERTIA. Airlines/MROs are risk-averse, ERP-entrenched (AMOS/TRAX/Ramco), and slow to trust a startup with safety-adjacent analytics. Even strong POC results may stall at procurement/airworthiness sign-off. Mitigation: enter as a low-risk overlay/companion with clear ROI, pursue MRO or lessor channel partners, and design for read-only integration first.
- COMMODITIZATION FROM ABOVE. If Skywise/Forge/AnalytX or an MRO ERP decides tires are worth a module, a neutral startup's wedge narrows fast. The current blind spot exists because tires aren't on the data bus - but that can change. Mitigation: build defensibility in the proprietary inspection-data models + inventory-optimization outcomes and airline relationships, not just the RUL algorithm.
- MARKET-SIZE FRAMING RISK. The aircraft tire market ($1.6-2.9B) and retread market (~$1.2M units) are modest and OEM-dominated; a software-only tire-RUL TAM is a slice of MRO IT spend, not the tire hardware market. Overstating TAM in the pitch invites pushback. Mitigation: size the opportunity as AOG-avoidance + inventory-optimization value (a cut of the ~$62B AOG problem and inventory budgets), not as a share of tire sales.

**Data / tech notes**

TECH FEASIBILITY: RUL is tractable on tabular inspection history (tread depth over landings, removal events) using survival/degradation models + gradient-boosting/time-series; academic tread-wear ML shows ~4.6-5.2% RMSE (1D-CNN) and ~99.7% wear-classification accuracy (image CNN), so an optional phone-photo CV enrichment is realistic. Tire cadence (~700-800 landings; some ~400) generates removal/inspection events fast enough to train per-fleet models quickly - a key advantage over slow-degrading parts like engines. The 5 POC capabilities map cleanly to: (1) RUL = degradation/survival model; (2) wear-to-limit date = extrapolate wear curve to 2mm limit; (3) wheel prioritization = rank by predicted time-to-limit + criticality; (4) inventory/maintenance planning = probabilistic demand forecast per station (mirrors Delta APEX >90% accuracy, ~20% inventory-cost cut benchmarks); (5) risk alerts = threshold + confidence-band breach on predicted AOG-driving wheels. INTEGRATION: treat AMOS/TRAX/Ramco as system-of-record (read inspection/component history, write prioritized recommendations); Bridgestone easytrack-style QR/serial tracking is the data-lineage backbone. DATA STANDARD: all timestamps UTC; wear-to-limit dates surfaced as calibrated ranges, not single points. KEY NUMBERS FOR PITCH: AOG $10-20K/hr narrowbody, $30-150K/hr widebody, up to $1.2M/day, ~$62B/yr industry-wide; ~90% of commercial aircraft on retreads; retread ~1.2M units/yr (2024)->~1.8M; aircraft tire market ~$1.6-2.9B (2024); PdM case studies: SmartLynx -57% AOG incidents/-49% downtime/~EUR840K/yr, Delta -95% maint cancellations, Azul 1-2 day->2-3 week prediction window, Airbus Skywise up to -15% unscheduled events. COMPETITIVE MAP: Bridgestone+JAL (algorithmic, OEM-locked, live) and Michelin/Safran PresSense (hardware/pressure, OEM-locked) are the direct threats; GE/Honeywell/Airbus/Boeing platforms omit tires (blind spot); AMOS/TRAX/Ramco track tires but don't forecast RUL (integration targets); Goodyear predictive is ground-fleet, not aircraft. DIFFERENTIATION VECTOR: vendor-agnostic + software-only + built on underused historical inspection records + wear/inventory-first (not pressure-first).

## POC scope, tech stack, architecture, demo narrative, and definition of done for an aircraft-tire RUL prediction pilot

**Key findings**

- All 5 pilot goals collapse to ONE model output plus arithmetic. Fit a regression that predicts tread depth as a function of cumulative usage, then derive everything else: RUL = model-projected landings until the wear limit; wear-to-limit date = today + (RUL / avg landings-per-day); priority = rank by lowest RUL and highest disruption cost; spares plan = count of wheels crossing the limit per week bucketed against on-hand stock; alerts = threshold rules on RUL/date. You do not need 5 models or a computer-vision pipeline for the POC — you need one wear-rate model and a scoring layer on top.
- This is a data-science demo, not a production service. The winning attribute is iteration speed and visual storytelling in front of maintenance stakeholders, not throughput, multi-tenant auth, or transactional integrity. That directly inverts what the org-default Kotlin/Micronaut + Next.js stack optimizes for.
- No real data exists yet ("historical inspection records are underused"), so a synthetic data generator is the critical path and the single biggest credibility risk. The generator must encode believable physics: per-position wear rates (nose vs main gear wear differently), aircraft-type differences, seasonal/route load effects, inspection-measurement noise, and censored/retired tires. If the synthetic data looks fake, the whole demo loses the room — so the generator deserves as much care as the model.
- A tree-based regressor (LightGBM) beats deep time-series models here. Inspection records are sparse, irregular, tabular, and low-volume per tire. Gradient-boosted trees handle small tabular data, mixed feature types, and missing values well, train in seconds, and — crucially for a stakeholder demo — expose feature importances you can narrate. LSTMs/Transformers are wrong for this data regime and would slow iteration.
- RUL is naturally expressed in landings (cycles), not days. Tires wear per landing, not per calendar day. Predict remaining landings-to-limit, then convert to a calendar date using each aircraft's recent landings-per-day utilization. This split makes the model physically honest AND makes the maintenance-planning and inventory views land, because planners think in dates and stock counts.

**Recommendations**

- Smallest lovable POC = one Streamlit app backed by one LightGBM wear-rate model over synthetic data, covering all 5 goals through five screens. Screen 1 Fleet Health Overview: fleet KPIs (wheels at-risk in 30/60/90 days, AOG-risk count) + a color-coded fleet/aircraft grid. Screen 2 Per-Wheel RUL & Wear Curve: measured tread-depth points, the fitted wear curve extrapolated to the wear limit with a confidence band, RUL in landings, and predicted wear-to-limit date. Screen 3 Priority Worklist: sortable table of wheels ranked by urgency score (RUL + disruption cost), with recommended action. Screen 4 Inventory / Spares Planner: weekly demand forecast (wheels crossing the limit per week) vs on-hand stock, flagging stock-out weeks and reorder points. Screen 5 Alerts Feed: rule-triggered cards (imminent wear-out, accelerating wear vs baseline, projected stock-out). That is the entire scope — resist adding real data ingestion, CV inspection, or auth.
- Pick Streamlit over FastAPI+React, and pick the Python DS stack over Kotlin/Micronaut + Next.js. Justification, made explicit: (a) Streamlit turns a pandas DataFrame + Plotly figure into an interactive screen in a few lines — no API contract, no frontend build, no state plumbing — so a 5-screen demo is days not weeks; (b) the model, features, scoring, and UI live in one Python process, so an insight discovered in a notebook reaches the screen the same afternoon (the core loop of a DS POC); (c) LightGBM/scikit-learn/pandas/Plotly are the native, best-in-class tools for this modeling work and have no first-class Kotlin equivalent — choosing Kotlin means either reimplementing ML or shipping a Python sidecar anyway; (d) Kotlin/Micronaut + Next.js is built for production concerns (typed multi-service APIs, DI, transactions, RBAC, scale) that a throwaway demo does not have, so its ceremony is pure tax here. Keep FastAPI+React only as the graduation path IF the POC is greenlit for production — do not pay that cost now. Recommendation: Streamlit for the POC, with a clean internal seam (a `scoring.py` module the UI imports) so the model/scoring layer could later be lifted behind FastAPI without a rewrite.
- Build the architecture as five decoupled Python modules in a linear pipeline so each is independently demoable and swappable: (1) `generate_data.py` — synthetic tire/inspection/utilization generator writing tidy CSV/Parquet (tires, inspections, aircraft-utilization tables); (2) `features.py` — per-inspection feature engineering (cumulative landings, days in service, wear rate since last inspection, position, aircraft type, load proxy, tread remaining); (3) `train.py` — LightGBM wear-rate/remaining-landings model with a persisted `model.pkl` and a held-out metric report; (4) `scoring.py` — pure functions: predict wear curve, compute RUL in landings, convert to wear-to-limit date, compute urgency score, roll up weekly spares demand, evaluate alert rules; (5) `app.py` — Streamlit UI importing `scoring.py`. Keep the model-to-UI boundary in `scoring.py` so the demo brain is testable and portable.
- Anchor the demo narrative on avoided AOG. Open on Fleet Health ("here is the whole fleet at a glance; 7 wheels go critical in the next 30 days"), drill into one wheel to show the wear curve extrapolating to its limit and a concrete date, jump to the Priority Worklist to show which wheels the team should touch first and why, switch to the Spares Planner to show week-3 runs short on this tire size so reorder now, then land on the Alerts Feed showing the disruption-risk alert that would have caught this before it became an AOG. Narrate the counterfactual: reactive today vs proactive with the tool. Seed the synthetic data so one wheel tells this exact story cleanly on every run.
- Define done concretely and cap effort. Definition of Done: all 5 screens live and navigable; each of the 5 pilot goals visibly demonstrated on-screen; synthetic generator produces a fixed seeded scenario that reproduces the demo story; model reports a held-out error metric (MAE on tread depth / on remaining-landings) plus a wear-curve fit plot; a scripted 10-minute walkthrough runs end-to-end without code edits; and a one-page README on assumptions, synthetic-data caveats, and the real-data path. Realistic effort for one competent DS/full-stack builder: ~1.5–3 weeks. Rough split: 2–3 days data generator, 1–2 days features+model, 1 day scoring layer, 3–4 days Streamlit UI/polish, 1–2 days demo scripting + buffer. Timebox: if the synthetic generator or model spills past its budget, ship simpler curves (per-position linear wear-rate fit) rather than cutting a screen — breadth across all 5 goals sells the pilot more than model sophistication.

**Risks**

- Synthetic-data credibility is the top risk. If wear curves, noise, or failure timing look invented, stakeholders discount every downstream number. Mitigation: ground the generator in publicly reasonable tire physics (per-position wear rates, tread-depth limits, landings-driven wear), add realistic measurement noise and censored/retired tires, and be explicit in the demo that data is synthetic and the pipeline is what transfers to real records.
- Demoing synthetic accuracy can mislead sponsors into over-trusting the model. On synthetic data the model will look great because it learns the generator you wrote. State plainly that reported error reflects synthetic conditions and that real-world validation on historical inspection records is the required next step before any maintenance decision is automated.
- Scope creep toward production and CV. The requirement lists computer-vision inspection and asset-health analytics as capabilities in play; pulling those into the POC would blow the timeline. Hold the line: CV and real-data ingestion are explicitly out of scope for the demo and belong in the post-greenlight phase.
- RUL-to-date conversion depends on assumed future utilization. Landings-per-day varies by route, season, and aircraft rotation; a wrong utilization assumption moves every predicted date. Mitigation: derive per-aircraft utilization from recent synthetic history, expose it as a visible assumption on the wear-curve screen, and let the demo show how changing utilization shifts the date.
- Streamlit's ceiling. It is ideal for this demo but not for many concurrent users, fine-grained auth, or complex multi-page state. That is acceptable for a POC; the risk is only if someone tries to push the Streamlit app toward production. Guard against it by keeping scoring logic in `scoring.py` so the production path is FastAPI + React re-using the same core, not a Streamlit rewrite.

**Data / tech notes**

Environment check: Python 3.14 is present on the machine; streamlit/lightgbm/plotly/pandas/sklearn are not yet installed but are a trivial `pip install streamlit lightgbm scikit-learn pandas plotly pyarrow` for the POC (no version conflicts observed). Recommended stack: Python 3.11+ (3.14 works), pandas + pyarrow for data, LightGBM (fallback scikit-learn GradientBoosting/RandomForest) for the wear-rate model, Plotly for interactive wear-curve/KPI charts, Streamlit for the 5-screen dashboard. Data model: three synthetic tables — tires (id, position [nose/main-L/main-R], aircraft_id, aircraft_type, install_date, initial_tread_mm, wear_limit_mm), inspections (tire_id, date, cumulative_landings, measured_tread_mm), aircraft_utilization (aircraft_id, date, landings_that_day). Model target: remaining tread as a function of cumulative landings (equivalently a per-tire wear rate); predict landings-to-limit, then date = today + landings_to_limit / recent_landings_per_day. Urgency score = weighted combination of (inverse RUL, disruption cost by aircraft/route criticality, spare availability). Alert rules (pure thresholds, no ML needed): RUL < N landings; wear-to-limit date within X days; wear rate accelerating > Y% vs the tire's own baseline; projected weekly demand > on-hand stock. Keep all tunables (wear limit, alert thresholds, planning horizon) in one config block so they are demoable as sliders. Architecture seam: UI imports pure functions from scoring.py so the same core can later sit behind FastAPI for a React production build — the org-default Kotlin/Micronaut + Next.js stack is the right target for a shipped product but the wrong tool for a fast DS demo, because it has no first-class ML libraries and its API/DI/build ceremony adds days of cost that a throwaway demo cannot justify. No file was written; this is a scoping recommendation only.
